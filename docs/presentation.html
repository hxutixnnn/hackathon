The Problem

Auditing a repo for vulns is slow, shallow, or both

Sequential scanners

Snyk / Semgrep / GHAS pattern-match. Miss logic bugs. Ignore intent.

Single-LLM scanners

One prompt, one pass. Token limit kills coverage past ~50 findings.

"Just use Codex CLI"

Nondeterministic. No cache. No benchmark. No way to prove accuracy.

15 attack angles × 10 file chunks × 4s sequential = 10 minutes per scan. Demo unusable.

The Solution

Paste a GitHub URL → 15 specialized auditors fan out across every file chunk → findings stream live

  • 15 attack angles per chunk: SQLi, XSS, authz, SSRF, secrets, deserialization, race, …
  • Workpool concurrency 20 — bounded parallelism, automatic retry
  • Live constellation UI — every node = one auditor, pulses on new finding
  • Convex live queries — zero polling, real-time stream to browser

90s vs 10 min sequential

Architecture

Convex orchestrates · Workpool bounds · OpenAI scores · Browser subscribes

                  │
                  └── live queries ── React (constellation, table, drawer)

No polling. No queues we run. Convex scheduler + Workpool do it all.

The Accuracy Moat

What a "Codex CLI + prompt engineer" cannot replicate over a weekend

Content-addressed cache

sha256(angle|model|prompt|chunk)

Re-scan same code = 0 LLM tokens. Hit rate shown live in StatsBar.

Embedding + union-find dedup

Cosine ≥ 0.85, gated by file+line overlap. Deterministic — pure math, not an LLM.

Scales past 1000 findings; current single-LLM reducers cap at ~50.

Juice-shop benchmark

Hand-curated truth set. Every scan scored: Precision · Recall · F1.
